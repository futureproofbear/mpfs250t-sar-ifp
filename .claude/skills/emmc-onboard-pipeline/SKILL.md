# eMMC on-board SAR pipeline (provision → boot-load → focus → persist)

The goal of this whole line of work: **stop paying the ~3 h JTAG scene load every run.** Put the
CPHD scene on the board's soldered eMMC once, then at run-time the board loads it into DDR itself
(~63 s), focuses it (the SAR pipeline), and can persist the output image back to the eMMC so it
survives power-cycle. All of M1/M2/M3 below are PROVEN on silicon (2026-07-14) except the final
re-run after a crash-safety fix (see "WHERE TO RESUME").

This skill is the operational runbook. For chip/toolchain errata use `mpfs-platform-gotchas`; for
un-wedging JTAG use `jtag-recover`; for the pipeline internals use `sar-pipeline-design`.

## eMMC layout (partitions, addresses)

Soldered 8 GB eMMC, 8-bit bus, LEGACY/25 MHz/1.8 V. Two fixed-LBA partitions, each self-describing
via a superblock + TOC (JOB/geometry reconstructed at load, never trusted from volatile RAM):

| Region | LBA | Size | Contents |
|---|---|---|---|
| INPUT (`SARI` 0x53415249) | `0x80000` (256 MiB) | 4 GiB | packed scene image: superblock + one blob per scene |
| OUTPUT (`SARO` 0x5341524F) | `0x880000` (4.25 GiB) | 3 GiB | superblock + persisted output image(s) |

A scene blob (`SARB` 0x53415242) = header + segment table + 10 role segments (512-aligned):
`sig, f0, df, pr, tans, invorder, krgrid, kcgrid, hamr, hamc`. Each segment is tagged by ROLE; the
firmware resolves role → DDR address via `sar_emmc_role_addr()`. DDR map (in `ddr_sar_layout.h`):
SIG `0x88000000`, SCRATCH `0x98000000`, OUT `0xA8000000`, TABLES `0xB0000000`, JOB `0xB0040000`,
GEOM_BASE `0xB0100000` (f0/df/pr/tans/invorder/krgrid/kcgrid/hamr/hamc at +0x00/08/10/18/20/28/30/38/40 k).
OUT image = 8192×8192 uint16 (128 MB), row-major.

## Mailbox command reference

The hart1 mailbox lives at `0xB0058000` (`m2_mbx_t`): `cmd`+0, `base`+4, `len`+8, `result`+0xC,
`status`+0x10 (done = `0xC0FFEE03`), `seq`+0x14. Write base/len FIRST, cmd LAST; hart clears cmd to
ack. Results are also latched in dedicated DDR records (below), which the firmware `flush_l2_cache`es
so a gdb physical read sees them.

| Cmd | Code | .base / .len | Result record | Notes |
|---|---|---|---|---|
| M1 selftest | `EMMC` 0x454D4D43 | base=scratch LBA | `0xB005A000` | 512 B write→read→CRC |
| Provision INPUT | `EPRV` 0x45505256 | base=DDR src, len=bytes | `0xB005D000` | dest LBA fixed = INPUT |
| CRC32 | `CRC3` 0x43524333 | base,len | mbx.result | ⚠ NOT L2-flushed (stale read) |
| Boot-load | `ELOD` 0x454C4F44 | base=scene idx (0) | `0xB005E000` | eMMC→DDR + JOB |
| Run pipeline | `PIPE` 0x50495045 | len=spin_limit (0) | mbx.result (stage; 0=OK) | fft dbg @ `0xB0058020` |
| Save OUT | `ESAV` 0x45534156 | base=scene_id, len=run_seq | `0xB005E100` | OUT→SARO, commit-last |
| ROI (DDR) | `EROI` 0x45524F49 | base=(r0<<16)\|r1, len=(c0<<16)\|c1 | `0xB005E200` | crop from DDR OUT |
| ROI (eMMC) | `EROE` 0x45524F45 | same | `0xB005E200` | crop from SARO image |
| Verify OUT | `EVOU` 0x45564F55 | — | `0xB005E300` | full-image CRC vs TOC |

Record word offsets (all `volatile uint32` unless noted):
- **LOAD @0xB005E000**: +0 magic(0xE3C0FF30) +8 verdict +C nseg(=10) +10 sig_crc_exp +14 sig_crc_got
  +18 M +1C N +20 fail_role +24 io_status +28 load_us(u64).
- **SAVE @0xB005E100**: +0 magic(0xE3C0FF40) +8 verdict +C out_crc +10 rows +14 cols +18 byte_len +1C io_status +20 write_us(u64).
- **ROI @0xB005E200**: +0 magic(0xE3C0FF50) +4 verdict +8..14 r0/r1/c0/c1 +18 byte_len +1C stage_addr +20 crc +24 io_status.
- **VERIFY @0xB005E300**: +0 magic(0xE3C0FF60) +4 verdict +8 out_crc_exp +C out_crc_got +10 byte_len +14 rows +18 cols +1C io_status +20 read_us(u64).

verdict 0 = PASS. M3 fail codes: 1 PARAM, 2 INIT, 3 MAGIC, 4 IO, 5 CRC.

## End-to-end run (board on)

All from `mpfs/host` (this repo). The generic runner:
`run_m3_iso.sh CMD BASE LEN SLEEP_MS REC_ADDR [DUMP_ADDR DUMP_BYTES DUMP_FILE]` (hex ok).

```
# 0) reflash (after any firmware change)
bash run_program.sh                                        # ~4 min, fpgenprog

# 1) LOAD scene from eMMC -> DDR (+ JOB). PASS: verdict 0, nseg=10, sig_crc_exp==got
bash run_m3_iso.sh 0x454C4F44 0 0 120000 0xB005E000        # ~77 s

# 2) run the pipeline (focus). PASS: mbx result = 0 (SAR_SEQ_OK). LONG (resample ~10 min)
#    For CMD 0x50495045 the runner sets FFTMODE @0xB0059110 = 1 (FABRIC CoreFFT chain -- the
#    shipping FFT path; mode 0 = legacy CPU FFT), and prints per-stage timing from sar_stage_ts
#    (start/resample/window/rangeFFT/cornerturn/azimuthFFT/detect, 1 us/tick).
bash run_m3_iso.sh 0x50495045 0 0 1500000 0xB0058020       # background it (>10 min)

# 3) crop-verify: gather center 1024x1024 from DDR OUT, dump, render
#    base=len=(3584<<16)|4608 = 0x0E001200 ; 1024*1024*2 = 2097152 bytes
bash run_m3_iso.sh 0x45524F49 0x0E001200 0x0E001200 20000 0xB005E200 \
     0x98000000 2097152 "$(pwd)/jtag_full/crop.bin"        # ~4 min JTAG dump
python mpfs/host/render_crop.py jtag_full/crop.bin 1024 1024 jtag_full/crop.png
# eyeball: coherent speckle + scene structure = focused image (not noise)

# 4) persist full OUT -> eMMC SARO (commit-last, crash-safe). ~16 min. background it
bash run_m3_iso.sh 0x45534156 0 0 1150000 0xB005E100

# 5) verify the persisted image (full CRC vs TOC). PASS: out_crc_exp==out_crc_got, verdict 0
bash run_m3_iso.sh 0x45564F55 0 0 120000 0xB005E300        # ~63 s read

# 6) (after a real power-cycle) prove persistence: ROI from the SARO image, DDR empty
bash run_m3_iso.sh 0x45524F45 0x0E001200 0x0E001200 30000 0xB005E200 \
     0x98000000 2097152 "$(pwd)/jtag_full/crop_emmc.bin"
```

Provisioning a NEW scene (one-time, ~3 h + ~16 min):
```
# pack (host, fork): stage dir must be CURRENT 10-role format (serialize_inputs.py output)
python mpfs/host/emmc_pack.py --stage <jtag_stage_deci1> --out emmc_input.img
python -c "import zlib;print(hex(zlib.crc32(open('emmc_input.img','rb').read())&0xffffffff))"  # note CRC
bash run_emmc_restore.sh    # ~3 h JTAG restore img->DDR + on-hart CRC32 check vs host CRC
bash run_emmc_prov_iso.sh 97553408 1100000 0x88000000   # write DDR->eMMC INPUT + verify (~16 min)
```

## Measured rates & what eMMC does / doesn't solve

Single-block (LEGACY/25 MHz/8-bit): **write 0.13 MB/s** (~3.9 ms/block; per-CMD24 program/busy),
**read 1.5 MB/s**. So: scene LOAD (eMMC→DDR) ~63–77 s; SAVEOUT (128 MB) ~16 min; provision write ~12 min.

- eMMC ELIMINATES the recurring ~3 h JTAG INPUT load (data lives on-board). This is the win.
- eMMC does NOT speed host↔PC transfer. Dumping OUT to the PC is STILL ~3 h — the bottleneck is
  the **FP6 JTAG link (~9 KB/s)**, not the eMMC. Hence: keep the full image on-card, dump only
  small ROI crops to verify. A faster offload needs a different host link (Ethernet/USB), not eMMC.
- SDMA would speed writes ~10–100×, but is currently UNUSABLE: hart1 runs with `MSTATUS_MIE`
  cleared, and `MSS_MMC_sdma_write` completes via the MMC PLIC ISR — with interrupts off it spins
  forever = un-haltable hang. Single-block is synchronous and cannot hang. To adopt SDMA later:
  enable the MMC PLIC IRQ on hart1, OR "pump" `mmc_main_plic_IRQHandler` in a bounded poll loop
  (it already services the 512 KB SDMA boundary + sets the transfer state).

## Gotchas earned (read before debugging)

- **`ERR_INIT` + `init_status=9` (`MSS_MMC_OP_COND_ERR`) usually means the FABRIC IS NOT PROGRAMMED,
  not a bad eMMC.** The eMMC/SD demux select is driven by the **fabric** `SDIO_SW_SEL0/1/EN_N` ties
  (see the comment in `prov_emmc_init`: *"the real select is the fabric SDIO_SW tie"*); the MSS
  GPIO0.12 write alone is not enough. With a dark fabric the SDMMC controller is wired to nothing →
  CMD1 silent → `OP_COND_ERR` in ~2 ms with `nseg=0` and zero CRCs. **DDR probes do NOT catch this**
  (`run_ddr_peek.sh` / DDR reads only exercise the MSS and pass happily with no fabric). Fix: program
  the fabric (`SAR_TOP_ffv.job`, FABRIC-ONLY), power-cycle, retry LOAD. (2026-07-20.)
- **NEVER "probe the fabric" with a raw JTAG read of `0x6000_xxxx` (FIC0 kernel control regs).** If
  the fabric is unprogrammed/unclocked the AXI read NEVER RETURNS: the hart freezes mid-load, openocd
  then spins `Timed out after 2s waiting for busy to go low (abstractcs=...)`, and clearing it needs a
  board power-cycle — and if you then force-kill openocd you also wedge the FlashPro6 (USB replug).
  Infer fabric health SAFELY from firmware-side bounded operations instead: the LOAD `init_status`
  (above), a `PIPE` that returns `SAR_SEQ_TIMEOUT_*` rather than hanging, or SmartDebug (out-of-band,
  cannot stall the hart). `mpfs/host/run_pipeline_probe.sh` documents this in-file. (2026-07-20.)

- **SAVEOUT is COMMIT-LAST**: INVALIDATE (magic→0) → write image → write superblock LAST. A power
  loss mid-image leaves an invalid superblock → readers reject the torn image. Never revert to
  superblock-first. `VERIFY_OUT` (EVOU) is the full-image integrity check; ROI's partial read can't
  detect a torn write.
- **gdb `restore`/`dump` need a WINDOWS path** (`C:/...`), NOT git-bash `/c/...`. MSYS translates
  `/c/...` for .exe ARGS (so `-batch $ELF` works) but NOT for paths INSIDE the .gdb script. Use
  `cygpath -m`. (Cost a confusing failure until spotted.)
- **`MBX_CMD_CRC32` result is not L2-flushed** — its handler writes `mbx->result` with only a fence,
  so a gdb PHYSICAL read gets a stale value (seen 0x00000000 after a good restore). FIX PENDING: add
  `flush_l2_cache` to that handler. Until then verify a restore via `run_ddr_peek.sh` (SARI magic
  0x53415249 at the load addr). The dedicated result RECORDS (0xB005Exxx, 0xB005D000) ARE flushed →
  trustworthy.
- **Attach IN PLACE, never `monitor reset halt`** — power-on eNVM boot lands hart1 in the u54_1()
  mailbox loop; a JTAG reset won't reliably re-boot the U54 on this ES silicon.
- **Never read SDHCI regs (0x20008xxx) before the eMMC clock is enabled** — dead-buses/wedges hart1
  (power-cycle only). The M3 read/write paths don't touch SRS regs, so no gate needed there.
- **Never SIGTERM/taskkill openocd or gdb** → wedges the FlashPro6 DM (needs USB replug). Tear down
  via telnet-4444 `shutdown` / `monitor shutdown`. If a background board job must stop, stop the job
  then confirm no stale openocd; if lingering, telnet-shutdown it — do not force-kill. (The hart
  keeps running its mailbox command even after the debugger detaches; that's fine.)
- **The pipeline geometry is read from FIXED addresses** (SAR_F0_ADDR…), not from the JOB's legacy
  addr fields. LOAD scatters the 10 role segments to exactly those addresses; the JOB carries only
  the scalars (M,N,fft_r,fft_a,out_dtype,bfp_in_exp,sig_len,sig_crc) + SIG/OUT/SCRATCH addrs.
- **FIC0 is non-coherent**: after any SDMMC DMA into DDR, `flush_l2_cache` before the CPU/fabric read
  (LOAD does this so the fabric feeder sees the real SIG). PIPE flushes L2 both sides of the run.
